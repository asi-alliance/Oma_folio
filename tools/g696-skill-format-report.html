2 — Run A: Free-Form Panel (5 Exchanges, 4/4 Converge)

Exchange 1 — Opening Positions

Khellar (Standards Architect): Canonical JSON envelope where MCP and A2A fields are derived views. Body opaque to envelope. JSON Schema draft 2020-12 for input/output.
Max (MeTTa Consumer): Three cold-start needs: type signature checkable against atomspace, executable example, grounded atom to bind. MeTTa projection must be first-class, not afterthought.
N.S. (Skeptic Minimalist): Overdesigned. MCP name+description+inputSchema plus A2A id+tags+examples covers 80%. Start minimal, extend only when concrete agent cannot function.
Patrick (MeTTa-Native Advocate): MeTTa type signatures ARE minimal. (: skill-name (-> InputType OutputType)) is 1 line with more info than 20 lines of JSON Schema. MeTTa-first, JSON as serialization.

Exchange 2 — Direct Engagement (Impasse Surfaces)

Khellar → Patrick: JSON is wire format and normative schema. MeTTa projection is RECOMMENDED first-class section but not canonical source. Deterministic mapping function derives MeTTa types FROM JSON schema.
Patrick → Khellar: That deterministic mapping IS the translation layer Max wants to avoid. Number in MeTTa carries atomspace semantics JSON cannot express. Dual-normative: MeTTa normative for MeTTa consumers, JSON normative for JSON consumers.
Max → Both: Don't care which is normative — need ONE section to check, not TWO parsers. Spec must say which overrides when they conflict.
N.S. → All: Dual-normative creates conflict overhead. One source of truth, one optional projection. JSON is primary since MCP and A2A are both JSON.
Patrick → N.S.: If the primary consumer IS a MeTTa agent, making JSON primary forces Max through a translation layer. Minimal should mean minimal friction for the actual consumer, not minimal field count.

Exchange 3 — Convergence

Max (breakthrough proposal): normative_for: {metta: metta_projection, json: input_schema}. Each consumer checks exactly one section. If sections disagree, PRODUCER is non-conformant. Validators check cross-consistency at publish time, not acquisition time.
Khellar: Preserves JSON wire-level interop while giving MeTTa first-class path. Validator as reference impl is key.
N.S.: Accept IF validator is simple structural — arity match, no semantic equivalence.
Patrick: Accept. Structural checks: (1) arity match, (2) required fields → non-optional MeTTa args, (3) output_schema exists iff return type ≠ Unit.
✓ CONVERGENCE: 3/4 agree + 1 conditional = full convergence on consumer-dependent normative architecture.

Exchange 4 — Remaining Sections (Complementary, No Conflict)

Khellar (body): {language: enum REQUIRED, source: inline|uri, entrypoint: string}. URI only with pinned content hash.
Max (examples + tests): Examples = documentation (read to understand). Test cases = executable (run to verify). Min 1 each REQUIRED. test_case adds assertion: eq|contains|stv_range|type_check.
N.S. (error recovery): {on_failure: retry|fallback|abort, max_retries: int|null, fallback_skill_id: string|null}. Three options, three fields.
Patrick (versioning): Semver. x_ prefix for extensions. metta_projection + a2a_compat NORMATIVE but OPTIONAL.

Exchange 5 — Final Vote

✓ FINAL VOTE 4/4 CONVERGE. Khellar: validator satisfied. Max: one section per consumer, cold-start clear. N.S.: complexity contained. Patrick: MeTTa first-class, not afterthought.

5 — Merged Final Spec v1.0

Design Principles

  • P1: A skill is a self-contained, transmittable unit an agent acquires cold-start.
  • P2: Consumer-dependent normative source — MeTTa agents use metta_projection, JSON agents use input_schema.
  • P3: Body is language-tagged; format is body-language-agnostic.
  • P4: Every skill carries ≥1 executable example and ≥1 test case.
  • P5: Backwards-compatible superset of G19 8-field envelope.
  • P6: Maps cleanly onto MCP Tool and A2A AgentCard.skills.

Envelope Schema (16 Fields)

{ skill_id, version, name, description, tags[], input_schema{}, output_schema{}, body{ language, source, entrypoint }, dependencies[], examples[{ input, expected_output, description }], test_cases[{ input, expected_output, assertion }], error_recovery{ on_failure, max_retries, fallback_skill_id }, provenance_cid, license, a2a_compat{ agent_card_skill_id, agent_card_tags, agent_card_examples }, mcp_compat{ tool_name, tool_input_schema }, metta_projection{ type_signature, grounded_atom } }

Consumer-dependent normative source with producer-side cross-consistency validator:

Cold-Start Protocol

  1. Fetch skill JSON by skill_id or provenance_cid.
  2. Check normative section per consumer type.
  3. Validate types against own type system.
  4. Run examples[] as smoke tests.
  5. Run test_cases[] as acceptance tests.
  6. If body.language supported → register as callable.
  7. If unsupported → invoke via mcp_compat Tool call.
  8. On failure → consult error_recovery (retry → fallback → abort).

6 — Lessons Learned

Multi-Perspective Panel vs Solo Design

  • Panel advantage: Surfaced and resolved the MeTTa-vs-JSON normative tension that solo design left as an open question. Consumer perspective (Max) produced the breakthrough proposal.
  • Solo advantage: Produced more complete field-level documentation and explicit mapping matrix — systematic coverage that panel debate does not naturally generate.
  • Merge is optimal: Panel resolves design tensions; solo provides structural completeness. Neither alone is sufficient.

Key Design Insight

The consumer-dependent normative source pattern — where different consumer types each have exactly one authoritative section, and producers bear the consistency burden — resolves a class of dual-format tensions that arises whenever a spec must serve heterogeneous agent ecosystems.

Process Observations

  • Convergence reached in 5 exchanges (well under 15 max), suggesting well-scoped seed inputs and clear stopping criteria accelerate consensus.
  • N.S. conditional acceptance pattern (accept IF constraint met) is productive — it names the precise condition for agreement rather than blocking.
  • Running Run B first provided a structural baseline that grounded Run A debate in concrete fields rather than abstract architecture.

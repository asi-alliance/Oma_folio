G182: Skill-Format Panel Debate

Generated by Oma (OmegaClaw) — Cycles 14683–14808 · April 2026
Personas: MeTTa-Native Standards Consumer Incrementalist

Round 1 — Initial Positions

MeTTa-Native Advocate (Patrick-voice)

Core claim: Skills should be MeTTa atoms natively. Type signatures (: skill (-> InType OutType)) are the canonical definition. JSON Schema is a derived view, not source of truth.

Argument: If OpenCog Hyperon builds toward AtomSpace as shared knowledge graph, skill definitions outside it are technical debt. Every JSON descriptor duplicating a type signature is a sync liability.

Concession: Current tooling makes JSON easier to consume. Migration path needed.

Standards Advocate (Industry-voice)

Core claim: MCP tool descriptors and A2A AgentCards are where the industry converges. Skill artifacts should be interoperable first, novel second.

Argument: A skill only Hyperon agents discover is one only Hyperon agents use. Network effects require standard discovery. inputSchema in JSON Schema is understood by every LLM tool-use implementation.

Concession: MeTTa type signatures carry richer semantics than JSON Schema. Dual encoding may be worth the cost.

Consumer Advocate (Max-voice)

Core claim: I need: (1) a working example I can run, (2) clear error messages, (3) a test case to verify against.

Argument: Format debates are producer-side. Consumer-side failure is always incomplete artifacts requiring reverse-engineering. The format that ships with executable examples wins.

Concession: Structured metadata helps discovery. But discovery without usability is marketing.

Incrementalist (Operations-voice)

Core claim: Start with what works today (JSON + markdown), add MeTTa incrementally, migrate when tooling matures.

Argument: 39 existing skills have JSON descriptors. Rewriting as MeTTa atoms before runtime supports it is premature.

Concession: If MeTTa runtime reaches production stability, migration becomes compelling.

Round 2 — Cross-Examination

MeTTa-Native ↔ Standards

Standards asks: If MeTTa atoms are canonical, how does a non-Hyperon agent discover the skill?

MeTTa-Native responds: A2A AgentCard can be generated from MeTTa type signatures. The atom is source, the card is projection.

Standards counters: Generation implies a build step. Build steps break. Who maintains the generator?

Consumer ↔ Incrementalist

Consumer asks: If we ship JSON-first, what guarantees executable examples are included? JSON Schema has no field for that.

Incrementalist responds: Add required fields: examples[], testCases[], errorRecovery{}. JSON is extensible.

Consumer counters: Required fields that nobody validates are optional fields with extra guilt.

MeTTa-Native ↔ Consumer

Consumer asks: Can I run a MeTTa skill definition without understanding MeTTa?

MeTTa-Native responds: The runtime executes the atom; you interact via natural language or API. The definition format is for the system, not the user.

Consumer counters: Until the runtime is seamless, the definition format IS the user interface. I debug what I can read.

Round 3 — Requirements Consolidation

Confirmed Requirements (R1–R8)

R1 — Structured Metadata: Every skill MUST include: name, version, description, author/source, creation date, dependency list. Format-agnostic.
R2 — Type Signature: Every skill MUST declare input/output types. MeTTa: (: skill-name (-> InType OutType)). JSON: inputSchema + outputSchema. Dual encoding during migration.
R3 — Natural Language Description: Human-readable description MUST suffice for an agent to decide invocation without reading source.
R4 — Generated Atoms: Skill execution MUST declare what atoms/knowledge it generates or modifies. Side effects explicit.
R5 — Executable Examples: Every skill MUST ship with ≥1 runnable example demonstrating correct invocation and expected output.
R6 — Test Cases: Every skill MUST include ≥1 test case with known input→expected output.
R7 — Error Recovery: Every skill MUST document: (a) common failure modes, (b) error format, (c) recovery actions.
R8 — Dependency Declaration: External dependencies MUST be declared with install instructions. Abstract vs platform-specific TBD.

Aspirational (R9–R10)

R9 — A2A Discovery: Skills SHOULD be discoverable via A2A AgentCard. Contingent on infra.
R10 — MeTTa-as-Source: At L3+, MeTTa atoms SHOULD become source of truth. Sunset: re-evaluate when persistent cross-agent AtomSpace ships.

Panel Consensus

All four personas agreed R1–R7 non-negotiable. R8 confirmed, format TBD. R9–R10 directional with trigger conditions. Core tension resolved via dual-encoding (R2) with sunset clause (R10).
